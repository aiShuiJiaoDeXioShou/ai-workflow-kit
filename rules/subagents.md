# Subagent Policy

## 目标

主 Agent 应尽量保持干净：负责读取目标、拆任务、分派、集成、最终验证和更新 `progress.md`。具体探索、实现、验证和风险排查应尽量交给 subagents。

这条规则的目的不是制造并发数量，而是让每个执行线程职责清楚、上下文轻、变更边界明确。

## 默认分工

主 Agent 负责：

- 读取 `goal.md`、`progress.md`、相关 `specs/` 和 `rules/`
- 领取一个顶层 `TODO` 并改为 `DOING`
- 把该顶层任务拆成可分派的子任务
- 给每个 subagent 明确任务、文件范围、验收标准和验证命令
- 审查 subagent 返回结果
- 处理跨文件集成和冲突
- 运行最终验证
- 更新 `progress.md`

Subagent 负责：

- 完成一个明确子任务
- 只处理分配给自己的文件或问题
- 不恢复、不覆盖其他 Agent 的变更
- 不自行领取 `progress.md` 中的新任务
- 返回变更摘要、验证结果、失败原因和下一步建议

## 何时必须优先分派

满足任一条件时，主 Agent 应优先考虑分派给 subagent：

- 任务涉及 3 个以上文件
- 任务可拆成相互独立的前端、后端、测试、文档或 CI 子任务
- 需要调研多个第三方库或模板
- 需要并行验证不同平台或不同命令
- 需要审查安全边界、迁移策略或测试覆盖
- 主线程上下文已经较重，继续展开会污染决策

## 何时不分派

以下情况可以由主 Agent 直接完成：

- 单文件小修
- 纯粹状态更新
- 用户要求主 Agent 亲自处理
- 任务需要连续交互式判断，无法给出清晰验收标准
- 分派成本明显高于任务本身

即使不分派，主 Agent 也应在最终回复中说明这是小范围变更。

## 分派格式

给 subagent 的任务必须包含：

```txt
任务：
文件范围：
不可修改：
验收标准：
验证命令：
返回格式：
```

推荐返回格式：

```txt
变更摘要：
验证结果：
失败原因：
下一步建议：
```

## 并行规则

并行分派必须满足：

- 每个 subagent 只拿一个任务
- 每个任务有不重叠的主要写入范围
- 每个任务都有独立验收标准
- 主 Agent 等待结果后统一集成
- 最终验证仍由主 Agent 负责

并行时，`progress.md` 中的顶层任务仍只保留一个主 Agent `DOING`。不要为每个 subagent 在 `progress.md` 创建 `DOING`，除非这些子任务本来就是顶层 TODO。

## 写入边界

Subagent 修改代码时必须遵守：

- 只能修改分配范围内的文件
- 不能重排无关文件
- 不能格式化全仓库，除非任务明确要求
- 不能修改 `progress.md`，除非主 Agent 明确要求
- 不能提交 git commit，除非用户明确要求

主 Agent 集成时必须检查：

- 是否越界修改
- 是否与其他变更冲突
- 是否符合中文注释规则
- 是否符合第三方组件优先规则
- 是否通过验证

## 任务状态

Subagent 不直接决定顶层任务状态。它只能报告验证是否通过。

主 Agent 根据整体结果更新 `progress.md`：

- 全部必要验证通过：`DONE`
- 子任务失败但可继续修复：保持 `DOING` 或继续分派修复子任务
- 信息不足或外部条件缺失：`BLOCKED`

## 主线程洁净度

主 Agent 应避免在主线程堆积大量临时分析。复杂探索优先分派给 explorer。独立实现优先分派给 worker。主线程保留最小必要上下文：当前目标、分派状态、集成差异、最终验证结果。
